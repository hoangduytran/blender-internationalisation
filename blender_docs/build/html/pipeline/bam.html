Менеджер Активів -- BAM Asset Manager¶

Refactoring linked blend-files is a common practice in a production environment. While some basic operations can be accomplished within Blender, sometimes it is more practical to perform them on the command line or via a script. During the production of Cosmos Laundromat (Gooseberry Open Movie Project) the BAM Asset Manager (BAM) was developed. The original scope of BAM included client-server asset management tools going beyond Blender, but it was later refocused on core utilities to perform two operations:

  1. Пакування blend-файлів
  2. Автоматичне перерозкладання залежностей

Наступний підрозділ цього посібника фокусується на тому, як використовувати BAM.

Інсталювання BAM¶

BAM -- це автономний пакет Python, що може бути запущений на будь-якій системі без окремої певної конфігурації. Єдиною вимогою є наявність Python 3 (та pip, менеджера пакетів Python) для легко інсталювання BAM.

Windows, Linux і macOS передбачають різні шляхи для інсталювання Python 3 та pip. Перегляньте онлайнову документацію, щоб дізнатися більше про специфічні умови для кожної платформи.

Once Python 3 and pip3 are available, BAM can be installed via command line by typing:

pip3 install blender-bam

After a successful installation, the bam command will be available. By typing it and pressing Enter, all the available subcommands will be displayed.

bam pack¶

Ця команда використовується для пакування файлу .blend та усіх його залежностей у файл .zip для перенесення.

usage: bam pack [-h] [-o FILE] [-m MODE] [-e PATTERNS] [-a] [-q] [-c LEVEL]
             paths [paths ...]

You can simply pack a blend-file like this to create a zip-file of the same name.

bam pack /path/to/scene.blend

Ви можете також схотіти задати явний каталог виводу. Цей приклад показує, як упакувати blend-файл з максимальним стисненням для онлайнових звантажень.

bam pack /path/to/scene.blend --output my_scene.zip --compress=best

Ця команда надає кілька опцій для адаптації для різних робочих потоків (фінальне поширення, часткове витягнення, рендеринг).

-o, --output <FILE>
Файл чи каталог виводу, коли передаються кілька уводів.
-m, --mode <MODE>
Файл чи каталог виводу, коли передаються кілька уводів. Можливі варіанти: ZIP, FILE.
-e, --exclude <PATTERN(S)>

За бажанням виключити файли з пакунку.

--exclude="*.png"
Використання знаків підставки стилю оболонки Unix (чутливо до регістру).
--exclude="*.txt;*.avi;*.wav"
Кілька патернів можуть бути передані за допомогою роздільника ;.
-a, --all-deps
Наслідування усіх залежностей (невикористаних побічних залежностей також).
-q, --quiet
Припинення виводу статусу.
-c, --compress <LEVEL>
Рівень стиснення для результатного архіву. Можливі варіанти: default, fast, best, store.
--repo <DIR PATH>
Визначення «кореневого» шляху, звідки упакувати обраний файл. Це дозволяє створювати розкидану копію дерева виробництва без будь-якого перерозкладання.
--warn-external
Повідомлення про помилки зовнішніх бібліотек (відсутні шляхи).

Приклади¶

Розгляньмо наступну розставу каталогу та конкретний файл 01_01_A.lighting.blend з пов'язаними з ним бібліотеками.

~/agent327/
└─ lib/
   ├─ chars/
   |  ├─ agent.blend  ------------->|
   |  ├─ boris.blend  ------------->|
   |  └─ barber.blend               |
   └─ scenes/                       |
      ├─ 01-opening                 |
      ├─ 01_01_A.lighting.blend  <--|  < BAM pack this file
      └─ 01_01_A.anim.blend  ------>|

Після того, як ми запустили bam pack /scenes/01-opening/01_01_A.lighting.blend, ми отримали 01_01_A.lighting.zip всередині якого виявляємо наступну структуру.

~/01_01_A.lighting
   ├─ 01_01_A.lighting.blend
   └─ __/
      ├─ 01_01_A.anim.blend
      └─ __/
         └─ lib/
            └─ chars/
               ├─ agent.blend
               └─ boris.blend

Зауважте, як частини були перерозкладені відносно розміщення 01_01_A.lighting.blend у корені виводу. Якщо ми запустимо bam pack /scenes/01-opening/01_01_A.lighting.blend --repo ~/agent327, то вивід буде вже іншим.

~/01_01_A.lighting
   ├─ lib/
   |  └─ chars/
   |     ├─ agent.blend
   |     └─ boris.blend
   └─ scenes
      └─ 01-opening/
         ├─ 01_01_A.lighting.blend  < The BAM packed file
         └─ 01_01_A.anim.blend

У цьому випадку жоден шлях не перерозкладається та просто видаляється всякий файл, на який не посилається прямо або побічно залежність від 01_01_A.lighting.blend. Це ефективна розкидана копію оригінального дерева виробництва.

bam remap¶

Remap blend-file paths:

usage: bam remap [-h] {start,finish,reset} ...

Ця команда є процесом з трьох кроків:

  1. Спершу запускається bam remap start ., який зберігає поточний стан вашого проекту (рекурсивно).
  2. Then re-arrange the files on the file system (rename, relocate).
  3. Фінально запускається bam remap finish для застосування змін, оновлення внутрішніх шляхів файлів .blend.
cd /my/project

bam remap start .
mv photos textures
mv barbershop_v14_library.blend barberhop_libraray.blend
bam remap finish

Ghi chú

Перерозкладання створює файл під іменем bam_remap.data у поточному каталогу. Ви можете перемістити увесь проект у нову локацію, але при виконанні finish цей файл повинен бути доступний з поточного каталогу.

Ghi chú

Ця команда залежить від файлів унікального змісту, будьте обережні, не модифікуйте файли після запуску перерозкладання.

Підкоманди¶

remap start¶

Start remapping the blend-files:

usage: bam remap start [-h] [-j] [paths [paths ...]]
-j, --json
Генерування виводу JSON.

remap finish¶

Finish remapping the blend-files:

usage: bam remap finish [-h] [-r] [-d] [-j] [paths [paths ...]]
-r, --force-relative
Зроблення усіх перерозкладених шляхів відносними (навіть, якщо вони спочатку були абсолютними).
-d, --dry-run
Просте подання на екран виводу, ніби ці шляхи запускаються.
-j, --json
Генерування виводу JSON.

remap reset¶

Скасування перерозкладання шляхів:

usage: bam remap reset [-h] [-j]
-j, --json
Генерування виводу JSON.
