Фізика «Загороджувач» -- Occluder Object Physics¶

Якщо об'єкт типу Occluder знаходиться між камерою та іншим об'єктом, то цей інший об'єкт не буде растеризуватися (обчислюватися для рендерингу). Він також відкидається, оскільки загороджений.

Ось демо blend-файл для ілюстрації деяких концепцій: BGE-Physics-Objects-Occluder.blend

  • Кострубатий, підподілений Куб під іменем "Cube".
  • Інший куб з іменем "Cube.BG" позаду площини з фізикою "Physics Type: Occluder".
  • Ще один куб з іменем "Cube.OffCamera" поза пірамідою Frustum огляду камери.

Тепер, подивимося, що станеться, у статистиці профілю для кожного з них, зробивши наступне (у такому порядку):

  1. Hit P as the scene is. It hums along at a fairly slow rate. On my system the Rasterizer step takes 130ms. The frame rate will finally jump up once the "Cube" object has completely moved out of the view frustum. It is as if the Occluder does not do anything while the Cube is behind it.
  2. Видаліть об'єкт "Cube.OffCamera" та зверніть увагу, що вже немає жодного підвищення швидкості. Це робота відкидання пірамідою огляду -- немає значення, чи такий об'єкт існує.
  3. Натисніть Z для перемикання на каркасний огляд. Зауважте, що у 3D View ви можете бачити "Cube.BG", але після натискання P, його там не буде.
  4. Make the "Occluder" object take up the whole camera's view with S X 5. You will see a huge leap in frame rate, since almost nothing is being Rasterized. On my system the Rasterizer step drops to 5ms.
  5. Спробуйте виконати з вимкненою World properties ‣ Physics ‣ Occlusion Culling. Це дасть сповільнення знову.
  6. Знову увімкніть World properties ‣ Physics ‣ Occlusion Culling та виконайте ще раз знову, щоб довести собі, що швидкість відновилася.
  7. Змініть для Occluder -- "Physics Type: Static". Зауважте, що сталося повернення до початкового сповільнення.
  8. Поверніть для нього тип "Physics Type: Occluder".
  9. Now make the "Occluder" invisible. The frame rate is back down to its original, slow rate.

Подробиці¶

Оскільки це стосується Фізики, то цей тип фізики є еквівалентом "No collision" для типу об'єкта Rigid. Причина, чому режим Occluder є взаємовиключним з іншими режимами фізики, полягає у тому, щоб акцентувати факт, що загороджувачі повинні бути спеціально призначені для такої цілі і не кожна сіть повинна бути загороджувачем. Однак, ви можете вмикати здатність Загородження на об'єктах з фізикою за допомогою Python та цеглин логіки. Дивіться (Link- TODO).

When an occluder object enters the view frustum, the BGE builds a Z-depth buffer from the faces of that object. Whether the faces are one-side or two-side is important: only the front faces and two-side faces are used to build the Z-depth buffer. If multiple occluders are in the view frustum, the BGE combines them and keeps the most foreground faces.

The resolution of the Z-depth buffer is controllable in the World settings with the "Occlusion Culling Resolution" button:

Стандартно роздільність є 128 пікселів для найбільшого розміру оглядвікна, тоді як роздільність іншого розміру задається пропорційно. Хоча 128 є дуже низькою роздільністю, це достатньо для цілей відкидання. Роздільність може бути підвищена до максимум 1024, але ціною значного навантаження CPU.

BGE проходить DBVT (Dynamic Bounding Volume Tree -- Дерево Динамічного Габаритного Об'єму) та для кожного вузла перевіряє, чи він повністю ховається загороджувачами, і якщо так, то відкидає цей вузол (та усі об'єкти, що він містить).

To further optimize the feature, the BGE builds and uses the Z-depth buffer only when at least one occluder is in the view frustum. Until then, there is no performance decrease compared to regular view frustum culling.

Рекомендації¶

Відкидання загороди є найбільш корисним, коли загороджувачі є великими об'єктами (будівлі, гори...), що ховають багато складних об'єктів у непередбачуваний спосіб. Однак, не хвилюйтеся надто про продуктивність: навіть, якщо ви використовуєте її неналежно, то зниження продуктивності буде лімітуватися через структуру цього алгоритму.

Існують ситуації, коли відкидання загороди не приносить ніякої користі:

  • Якщо загороджувачі є малими та не ховають багато об'єктів.

    У такому випадку відкидання загороди просто уповільнить CPU.

  • Якщо загороджувачі є великими, але ховають прості об'єкти.

    У цьому випадку вам кращі посилати такі об'єкти на GPU.

  • Якщо загороджувачі є великими та ховають багато складних об'єктів, але дуже передбачуваним способом.

    Приклад: будинок, наповнений складними об'єктами. Хоча відкидання загороди буде працювати добре у такому випадку, однак ви отримаєте кращу продуктивність, втіливши спеціальну логіку, що буде ховати/знеховати об'єкти; наприклад, робивши об'єкти видимими тільки тоді, коли камера входить у такий будинок.

  • Occluders can be visible graphic objects but beware that too many faces will make the Z-depth buffer creation slow.

    Наприклад, територія не є добрим кандидатом для загородження: надто багато граней та надто багато перекривань. Загороджувачі можуть бути невидимим об'єктами, поміщеними всередині складних об'єктів (наприклад, «у стінах» будівлі зі складною архітектурою). Загороджувачі можуть мати «діри», крізь які ви будете бачити об'єкти.
